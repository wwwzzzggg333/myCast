# myCast：iPhone 投屏到 Windows 设计规格

**日期：** 2026-07-20  
**状态：** 待用户确认后进入实现计划  
**范围：** v1 只做画面实时显示；声音、录制、反向控制为后续增强

## 1. 背景与目标

在 Windows 电脑上查看 iPhone 屏幕。用户环境已具备：

- 电脑与 iPhone 同一局域网
- USB 已连接

**成功标准（v1）：**

- 打开桌面 App 后，可通过 USB 或 Wi‑Fi 看到 iPhone 画面
- 断连、重连可预期；失败时有可读提示
- USB 路径在设备已信任前提下，通常约 10 秒内出画（不含用户在手机上操作信任的时间）
- Wi‑Fi 路径：本机可被控制中心「屏幕镜像」发现并成功镜像

**明确不做（v1）：**

- 系统声音同步
- 录屏 / 截图
- 鼠标键盘反向控制 iPhone
- 多设备同时投屏
- 跨网段 / 云投屏

## 2. 产品决策摘要

| 项 | 决策 |
|----|------|
| 核心用途 | 只看画面 |
| 连接方式 | USB + Wi‑Fi 都支持；先保证能看，再打磨体验 |
| 形态 | Windows 桌面窗口 App |
| 声音 | 后续增强 |
| 实现策略 | 尽快稳定出画，合理依赖成熟开源能力，不自研协议 |

## 3. 架构

**myCast** = Windows 桌面壳 + 两个可替换投屏后端，经统一 Session 接口协调。

```text
┌─────────────────────────────────────┐
│  myCast Desktop (UI)                │
│  连接 / 断开 / 通道选择 / 画面窗口   │
└──────────────┬──────────────────────┘
               │ 统一 Session 接口
       ┌───────┴───────┐
       ▼               ▼
  USB Backend     AirPlay Backend
  (QuickTime 流)  (AirPlay 接收)
       │               │
       └───────┬───────┘
               ▼
        视频帧 → 播放器表面显示
```

- **UI 层**：设备列表、连接状态、错误提示、全屏与保持比例显示
- **Session 层**：统一 `start / stop / status`，屏蔽 USB / Wi‑Fi 差异
- **USB 后端**：检测已信任设备 → 拉起屏幕流 → 交给播放器
- **AirPlay 后端**：本机启动接收服务 → 用户从 iPhone 控制中心镜像
- **技术倾向**：桌面壳用 Tauri 或 Electron（实现计划中最终选定）；协议层包装已验证开源能力，不自写 AirPlay / QuickTime 协议

## 4. 组件

| 组件 | 职责 | 依赖 |
|------|------|------|
| App Shell | 主窗口、菜单、全屏、设置入口 | UI 框架 |
| Device Panel | 列出 USB 已连接设备；Wi‑Fi 显示「等待镜像」状态 | Session |
| Session Manager | 统一启动/停止；同一时间只跑一个活动会话 | USB / AirPlay Backend |
| USB Backend | 枚举设备、检查信任、启动屏幕流、上报断连 | 成熟 USB/iOS 工具链 |
| AirPlay Backend | 启停接收端、暴露本机名称、上报连接事件 | AirPlay 接收组件 |
| Video Surface | 渲染画面、等比缩放、黑边处理 | 解码/播放管线 |
| Status / Errors | 将底层失败映射为可读提示 | 各 Backend |

**边界原则：**

- UI 不直接碰协议细节
- Backend 只产出「流已就绪 / 失败原因」，不画界面
- Session Manager 是唯一协调者，避免 USB 与 AirPlay 同时抢画面

## 5. 主流程

### 5.1 USB 投屏

1. 用户插入 iPhone → App 检测到设备
2. 若未信任：提示在手机上点「信任」，并在需要时提示安装/检查 Apple 设备支持组件
3. 用户点「开始投屏」→ Session Manager 选择 USB Backend
4. Backend 建立屏幕流 → Video Surface 显示
5. 拔线或点「断开」→ 停止会话，回到空闲态

### 5.2 Wi‑Fi（AirPlay）投屏

1. 用户选择「无线投屏」→ App 启动 AirPlay 接收，显示本机名称（默认 `myCast`）
2. 用户在 iPhone：控制中心 → 屏幕镜像 → 选中该名称
3. 收到连接 → Video Surface 显示
4. 手机停止镜像或 App 点「断开」→ 停止接收；可配置为保持服务待命或完全关闭

### 5.3 通道选择规则（v1）

- 用户手动选择 USB 或 Wi‑Fi
- USB 可用时界面优先提示 USB（通常更稳），但不自动打断进行中的 AirPlay 会话
- 同一时间只允许一个活动会话

## 6. 错误处理与边界

| 场景 | 用户提示 | App 行为 |
|------|----------|----------|
| 手机未点「信任」 | 明确提示去手机确认信任 | USB 会话失败，提供重试 |
| 缺少 Apple 相关驱动/服务 | 提示安装 iTunes 或 Apple 设备支持组件，并给出说明 | 不静默失败 |
| AirPlay 端口/名称冲突 | 提示更换名称或关闭占用程序 | 启动失败后可改配置重试 |
| 防火墙拦住发现 | 提示放行 myCast / 相关组播规则 | Wi‑Fi 发现失败时给出排查步骤 |
| 投屏中途断连 | 状态「已断开」，画面清空或短暂冻结后提示 | 清理会话，不残留僵尸进程 |
| 后端进程崩溃 | 「投屏异常退出」+ 重试 | Session Manager 回收子进程 |

## 7. 测试与验收

### 7.1 手动验收（v1 必过）

- USB：信任后一键投屏，画面连续；旋转后仍正常显示
- Wi‑Fi：控制中心能搜到 `myCast` 并成功镜像
- 断开/重连：反复 5 次无僵尸进程、无端口占用残留
- 失败可读：未信任、驱动缺失、断线时提示正确

### 7.2 自动化

- Session Manager 单元测试：状态机 `idle → connecting → streaming → stopped`
- Backend 适配层 mock：模拟成功/失败/崩溃，验证提示文案，不依赖真机
- 真机 E2E 不强制进入 CI（需要实体 iPhone）
